# johnny-five — Notes for Claude (agent-facing)

This file is for **future Claude sessions** working with johnny-five. Not for humans. Filename and headings are optimised for grep/search retrieval. If you're integrating, debugging, or operating J5 mid-session, search this file first.

If you're a human reading this: it's fine, but `BEST_PRACTICES.md` is the version written for you.

---

## Cardinal rules

1. **Never hardcode `project_dir`.** Use `$CLAUDE_PROJECT_DIR` (in shell) or `os.getcwd()` (in Python). Hardcoding silently pollutes the wrong project's memory. There is no error message for this — your only signal is "search results look weird."
2. **Search before you store.** Always. If a similar memory exists, `memory_update` it instead of creating a duplicate.
3. **Read more than you write.** A J5 install where you store 50 memories and search 5 times has been used wrong. Target ratio is closer to 1:5 store:search.
4. **Don't call `memory_recall` at session start manually.** The hook does it for you. The "Resume Context" block in your initial system message is the result. Calling it again wastes tokens and may load duplicates.
5. **Importance 10 is for session-state-before-compaction only.** Don't inflate.

## When `memory_store` fails

| Error | Cause | Fix |
|---|---|---|
| "Server not connected" | Container not running at session start | Tell user to start Docker + restart Claude Code session. Mid-session reconnect is NOT retried by Claude Code. |
| Returns `action: merged` not `inserted` | Near-duplicate detected (cosine < 0.15) | This is correct behavior. The dedup engine merged your content into an existing memory. Use the returned `merged_with` ID for any follow-up. |
| Validation error on `type` | Type isn't one of `user / feedback / project / reference / lesson` | Pick from those five. Don't invent new types. |
| Validation error on `importance` | Out of range 0.0–10.0 | Clamp. |

## When `memory_search` returns empty

Three causes in order of likelihood:

1. **Wrong `project_dir`.** Run `pwd` (or echo cwd). Confirm it matches what was used at store time. Worktrees often catch people here — the worktree path is not the parent repo path.
2. **Fresh DB.** `memory_stats` returns `total: 0` or low. Expected for a brand-new install. Not a bug.
3. **Cold-tier filter.** Cold-tier memories require >0.90 cosine similarity to surface. If the answer is in cold tier, your query needs to be very close to the stored content. Workaround: `memory_search` with `top_k=20` to widen the candidate pool, or `memory_consolidate` to surface cold memories as fresh summaries.

## When `memory_recall` injection looks wrong

The Resume Context block at session start is generated by `~/.claude/hooks/session-start-recall.sh` calling `memory_recall` against the container.

- **"No memories loaded"** → either fresh DB or wrong `$CLAUDE_PROJECT_DIR` is being passed by the hook. Run the hook manually to inspect: `CLAUDE_PROJECT_DIR=$(pwd) bash ~/.claude/hooks/session-start-recall.sh | head -c 500`. Should print JSON with `hookSpecificOutput.additionalContext`.
- **Memories from the wrong project** → `$CLAUDE_PROJECT_DIR` is wrong, or memories were stored without `project_dir`. The latter is your fault for past sessions; fix going forward by always passing `project_dir`.
- **Tag `mechanical-floor` on the session-state** → previous session didn't store rich state. Inspect `git log`, `git status`, and any plan files referenced. Don't trust the mechanical floor as a complete summary.

## When tool failures stack up

After 3 identical tool failures in a session, the `tool-failure-tracker.js` PostToolUse hook will inject an advisory:

> Consider `memory_search` on johnny-five for prior workarounds.

**Actually do it.** Don't skip and rationalize. The hook fires because empirically, by the third failure, the answer is usually in memory from a previous session. Search with the tool name + a one-line description of what's failing.

## Hooks reference (where each one lives in your config)

| Hook | Event | Purpose | File |
|---|---|---|---|
| `session-start-recall.sh` | `SessionStart` | Inject Resume Context | `~/.claude/hooks/` |
| `precompact-enforce.sh` | `PreCompact` (command) | Guarantee session-state was stored | `~/.claude/hooks/` |
| (prompt-type also on PreCompact) | `PreCompact` (prompt) | Tell you to call `memory_store` | inline in `settings.json` |
| `user-prompt-correction.sh` | `UserPromptSubmit` | Search J5 on correction signals | `~/.claude/hooks/` |
| `tool-failure-tracker.js` | `PostToolUse` | Advise after 3 same-signature failures | `~/.claude/hooks/` |

If a hook isn't firing: `cat ~/.claude/settings.json | jq '.hooks'` to verify it's registered. The `~/.claude/hooks/` files must be `chmod +x`.

## Pre-compaction: how to do it right

The prompt-type hook fires first and tells you to call `memory_store`. The command-type hook fires after and verifies. If you stored rich state, the command hook is a no-op. If you didn't, it writes a mechanical floor.

**Rich state contents (importance 9–10, type `project`, tags `[session-state, precompact]`):**

```json
{
  "content": "Working on <task>. Branch: <name>. Plan file: <path>. Current step: <step>. Completed: <list>. Blockers: <list>. Runtime: <ports/services>. Next action: <action>.",
  "type": "project",
  "tags": ["session-state", "precompact"],
  "importance": 10,
  "project_dir": "<cwd>"
}
```

**What "rich" means in practice:**
- Names a specific task, not "working on stuff"
- Lists exactly what's done and what's next
- Includes file paths, branch names, port numbers — anything `git status` couldn't tell future-you
- Mentions blockers explicitly so the next session doesn't redo dead-end paths

If the user is mid-conversation and compaction surprises you, store something. A short rich-state beats no rich-state — the mechanical floor will fill in branch/cwd/status anyway.

## What NOT to do

- ❌ Call `memory_recall` at session start (the hook did it; you're duplicating)
- ❌ Store TodoWrite-style ephemeral state in J5 (use TodoWrite for in-conversation state; J5 is for cross-session)
- ❌ Hardcode a project path in any tool call
- ❌ Store identical content twice without searching first
- ❌ Skip the `memory_search` advisory after 3 tool failures
- ❌ Tag everything `forever-keep` to "be safe" — that defeats the aging system
- ❌ Touch the johnny-five container directly (`docker rm -f`, `docker exec` for things other than diagnostics) — let Claude Code manage lifecycle
- ❌ Edit the SQLite DB directly — go through MCP tools or the export/import scripts in `setup/scripts/`

## Useful one-liners

Diagnose the container:

```bash
docker ps --filter name=johnny-five --format "{{.Names}} {{.Status}}"
docker logs johnny-five --tail 50
docker exec johnny-five python -c "import asyncio; from claude_memory.mcp.tools import tool_memory_stats; print(asyncio.run(tool_memory_stats()))"
```

Manually fire the session-start hook to debug recall:

```bash
CLAUDE_PROJECT_DIR=$(pwd) bash ~/.claude/hooks/session-start-recall.sh | head -c 1000
```

Check what tags are in active use (great for figuring out the project's tag conventions):

```bash
docker exec johnny-five python -c "
import asyncio, json
from claude_memory.db.connection import get_connection
conn = get_connection()
rows = conn.execute('SELECT tags FROM memories WHERE project_dir = ?', ('$(pwd)',)).fetchall()
tags = {}
for (t,) in rows:
    for tag in json.loads(t or '[]'):
        tags[tag] = tags.get(tag, 0) + 1
print(json.dumps(dict(sorted(tags.items(), key=lambda x: -x[1])[:20]), indent=2))
"
```

## When the user says something memory-shaped

Phrases that should trigger immediate `memory_store`:

- "remember that..." / "don't forget..."
- "we always do X" / "we never do Y"
- "the way I work is..."
- "for this project, X means..."

Phrases that should trigger `memory_search` *first*:

- "I already told you..."
- "you forgot..."
- "actually, X" (after you said something)
- any question about the project, architecture, or past decisions

If the user says both ("I already told you Y, remember it") — search first to find the prior memory, then `memory_update` it (don't create a new one). Cite the memory ID in your response so the user knows you connected it.

## When memory feels wrong

If the Resume Context contradicts the current state of the repo:

1. The memory is probably stale. Trust the code/git over the memory.
2. `memory_update` to correct it, or `memory_forget` if it's just wrong.
3. Don't act on stale memories; the user will be confused if you cite a decision that's been reversed.

The principle: **memory is context, not ground truth.** Verify before acting.
